# Dictator

MLX Whisper Dictation для macOS на Apple Silicon.

Офлайн-диктовка для macOS на Apple Silicon. Приложение живет в строке меню, записывает звук с микрофона, расшифровывает речь через MLX Whisper и вставляет результат в активное поле ввода.

Проект ориентирован на локальный запуск на Mac и упаковывается в `.app` через `py2app`.

Это репозиторий, который можно отдать другим владельцам MacBook Air или Pro на M1, M2, M3 и новее: они смогут клонировать код, собрать `.app` у себя и получить локальную диктовку на устройстве через MLX без облака.

## Требования пользователя к приложению

Ниже зафиксированы текущие продуктовые требования к приложению.

- Приложение должно работать как menubar-приложение без отдельного основного окна.
- Целевая платформа: macOS на Apple Silicon.
- Распознавание должно выполняться локально через MLX Whisper.
- Основной пользовательский сценарий: запись запускается из menu bar или по глобальному хоткею, затем результат вставляется в активное поле ввода.
- Вставка текста выполняется через настраиваемую цепочку методов: CGEvent Unicode, Accessibility API или буфер обмена с восстановлением. Если ни один метод не сработал, текст остаётся доступен в подменю «📋 История текста».
- Все распознанные тексты автоматически сохраняются в историю (до 20 записей), доступную из меню приложения.
- В меню должны отображаться:
	- текущий статус приложения;
	- текущая модель;
	- текущий хоткей;
	- максимальная длительность записи;
	- статусы критичных разрешений macOS.
- Интерфейс приложения, уведомления и документация должны быть на русском языке.
- Обновление приложения должно оставаться совместимым со сценарием, когда пользователь просто перетаскивает новую версию `.app` из `dist` в `Applications`.

## Требования пользователя к агенту

Для этого репозитория также зафиксированы ожидания к агенту разработки.

- Агент должен писать на русском языке.
- Агент должен учитывать, что пользователь обновляет приложение через drag-and-drop из `dist` в `Applications`.
- Агент не должен считать LaunchAgent главным способом эксплуатации приложения.
- При проблемах с хоткеями или вставкой агент должен проверять не только код, но и слой разрешений macOS.
- При изменениях в поведении приложения агент должен актуализировать документацию.
- Если автоматическая вставка недоступна, агент должен сохранять надежный fallback через историю текста и, при включённом методе, через буфер обмена с восстановлением.
- При каждом добавлении новой функции или изменении поведения агент должен дописывать или обновлять тесты и запускать их вместе с coverage.

Подробные правила для агента вынесены в [AGENTS.md](AGENTS.md).

## Структура кода

Текущая структура проекта соответствует Clean Architecture:

```text
src/
├── domain/          # типы, константы, порты и чистые правила
├── use_cases/       # сценарии записи, транскрибации, LLM, настроек и хоткеев
├── adapters/        # menu bar UI, overlay и hotkey dialog
├── infrastructure/  # macOS runtime, MLX, persistence, text input, hotkeys
└── app.py           # тонкий координатор DictationApp
main.py              # composition root
```

Ключевое правило: `domain` не знает о runtime, `use_cases` не импортируют `infrastructure`, а concrete macOS/MLX/NSUserDefaults зависимости собираются только в `main.py`.

## Как это работает

1. Приложение запускается как приложение в строке меню.
2. По глобальному хоткею начинается запись с микрофона.
3. После остановки записи аудио передается в `mlx_whisper.transcribe(...)`.
4. Распознанный текст сохраняется в историю (подменю «📋 История текста»).
5. Текст вставляется через цепочку включённых методов: CGEvent Unicode → AX API → Буфер обмена (Cmd+V).
6. Если все включённые методы не сработали, текст доступен в истории; буфер обмена пользователя не затрагивается.

### Поведение после sleep/wake macOS

- При выходе Mac из режима сна приложение автоматически обновляет список микрофонов.
- Если выбранный микрофон сменил системный индекс, приложение пытается найти его заново по имени.
- Если выбранный микрофон временно пропал, приложение переключается на системный микрофон по умолчанию и показывает уведомление.
- Глобальный хоткей после пробуждения также переинициализируется автоматически.
- Если sleep прервал активную запись, текущая сессия отменяется, чтобы не зависать в состоянии «идёт запись».

🍎 Apple Silicon и Neural Engine
Apple M-серии (M1/M2/M3/M4) содержат три вычислительных блока: CPU, GPU и Neural Engine (ANE) — выделенный чип для ML-инференса, способный выполнять до 18 TOPS (M3). MLX — это фреймворк Apple, спроектированный специально для этой архитектуры. Ключевое отличие: unified memory — CPU, GPU и ANE делят одно адресное пространство, и данные не копируются между ними, а просто «переключаются» zero-copy. Это значит, что 500 Мб модели Whisper загружаются в память один раз и сразу доступны для инференса на GPU. На обычных компьютерах те же данные пришлось бы сначала скопировать из RAM в VRAM — это задержка и двойной расход памяти. MLX Whisper использует эту архитектуру на полную: модель large-v3-turbo (809M параметров) на M3 транскрибирует 30 секунд речи за ~2–3 секунды, полностью локально, без единого сетевого запроса.


## Быстрый старт для владельцев Mac M-серии

Если не хотите собирать приложение самостоятельно, используйте готовый релиз.

1. Откройте страницу [Releases](https://github.com/zagitov/mlx-whisper-dictation/releases/latest).
2. Скачайте архив вида `Dictator-macos-apple-silicon-vX.Y.Z.zip` из последнего релиза.
3. Распакуйте архив: внутри будет `Dictator.app`.
4. Перетащите `Dictator.app` в `Applications`.
5. Запустите приложение и выдайте ему `Microphone`, `Accessibility` и при необходимости `Input Monitoring`.

Важно: для установки на обычную машину используйте именно архив из GitHub Releases. Локальная alias-сборка через `py2app -A` подходит только для разработки на том же Mac и будет падать на другой машине или вне текущего checkout.

Если релиза ещё нет или нужна локальная разработка, используйте сценарий сборки ниже.

Если цель простая: быстро поднять рабочее приложение на своем Mac, используйте такой сценарий.

1. Установите Homebrew, если его ещё нет.
2. Поставьте `portaudio` и `uv`.
3. Зафиксируйте Homebrew Python 3.11 для проекта.
4. Установите зависимости и dev-инструменты через `uv`.
5. Соберите `.app`.
6. Перенесите `.app` в `Applications`.
7. Выдайте приложению `Microphone`, `Accessibility` и при необходимости `Input Monitoring`.
8. Запустите приложение и пользуйтесь через хоткей.

Команды:

```bash
brew install portaudio
brew install uv
git clone <your-fork-or-repo-url>
cd mlx-whisper-dictation
uv python pin 3.11.14
uv sync --dev
uv run python setup.py py2app -A
open "dist/Dictator.app"
```

Приложение не использует облачную диктовку macOS и не отправляет звук во внешний сервис. Расшифровка идет локально на машине.

## 🔒 Приватность

Всё распознавание происходит на вашем устройстве. Приложение не отправляет аудио, текст или метаданные в интернет. В рантайме нет ни одного сетевого запроса — ни HTTP-клиента, ни сокетов, ни телеметрии.

- **Модель** скачивается один раз из Hugging Face при первом запуске и дальше работает полностью офлайн.
- **Аудио** обрабатывается в оперативной памяти и не сохраняется на диск, если не включён режим диагностики.
- **Текст** попадает в активное поле ввода через выбранный метод вставки. Буфер обмена затрагивается только при включённом методе «Буфер обмена (Cmd+V)», и в этом случае предыдущее содержимое буфера автоматически восстанавливается.
- **Логи** пишутся локально в `~/Library/Logs/whisper-dictation/` и содержат только техническую информацию о работе приложения.
- **Диагностические записи**: если диагностика включена, в `~/Library/Logs/whisper-dictation/recordings/` могут временно храниться WAV-файлы с голосом. Логи, диагностические аудио/JSON и история сообщений живут не дольше 24 часов.

Исходный код открыт — можно убедиться самостоятельно: `grep -r "http\|socket\|requests\|urllib" main.py` не вернёт ни одного совпадения.

## Локальная сборка приложения

Надежный локальный путь для этого проекта сейчас такой: собрать alias `.app` через `py2app -A`. Это обычный `.app`, но он использует текущее окружение проекта вместо полной standalone-заморозки всех зависимостей.

Такой alias `.app` нельзя использовать как релизный артефакт и нельзя распространять как готовое приложение для других пользователей. Он ссылается на локальный Python, `.venv` и исходники в текущем checkout.

```bash
brew install portaudio
brew install uv
uv python pin 3.11.14
uv sync --dev
uv run python setup.py py2app -A
```

После сборки приложение появится здесь:

```bash
dist/Dictator.app
```

Запуск:

```bash
open "dist/Dictator.app"
```

Обновление установленной версии, которое использует текущий пользователь:

1. Соберите новую версию в `dist`.
2. Перетащите `dist/Dictator.app` в `Applications` с заменой.
3. При необходимости заново проверьте разрешения в `Privacy & Security`.

## GitHub Actions и релизы

В репозитории настроен workflow [build-macos-app.yml](.github/workflows/build-macos-app.yml), который делает две вещи:

- на каждом push в `main` или `master` и при ручном запуске собирает standalone-версию `Dictator.app` и прикладывает zip-архив как GitHub Actions artifact;
- на push тега вида `v*` дополнительно создает GitHub Release и прикрепляет тот же архив как release asset.

Workflow дополнительно валидирует, что release bundle собран именно как standalone, а не как alias `.app` для разработки.

Практический сценарий публикации новой версии:

1. Обновите версию в `pyproject.toml` и `setup.py`.
2. Закоммитьте изменения в `main`.
3. Создайте и отправьте тег, например `v0.1.0`.
4. Дождитесь завершения workflow в GitHub Actions.
5. Проверьте, что в релизе появился архив `Dictator-macos-apple-silicon-v0.1.0.zip`.

Важно: GitHub не хранит `.app` как один файл, потому что `.app` на macOS — это директория bundle. Поэтому для скачивания публикуется zip-архив, внутри которого лежит готовый `Dictator.app`.

Если нужен запуск с выводом логов в терминал:

```bash
./dist/Dictator.app/Contents/MacOS/Dictator
```

Команда `uv sync --dev` ставит не только runtime-зависимости, но и инструменты для сборки и проверки кода, включая `py2app`, `modulegraph` и `ruff`.

## Почему именно Python 3.11

- В проекте зафиксирована `.python-version = 3.11.14`.
- Homebrew Python 3.11 является framework build, а это заметно лучше работает с `py2app` на macOS.
- Ранее `pyenv` Python 3.12 без framework давал менее пригодный результат для app bundle.

`uv python pin 3.11.14` удерживает проект на нужной версии Python и упрощает воспроизводимую настройку на других машинах.

## Self-check цикл

Чтобы не гонять линт, тесты и сборку вручную по отдельности, используйте единый сценарий:

```bash
uv run python scripts/selfcheck.py
```

Что делает команда по умолчанию:

- запускает `ruff check .`
- запускает основной набор pytest-тестов
- не трогает slow и hardware тесты
- не собирает `.app`, если это не нужно

Если нужно контролировать покрытие тестами:

```bash
uv run python scripts/selfcheck.py --coverage
```

Что добавляет этот режим:

- считает coverage по проекту;
- показывает `term-missing` отчёт по непокрытым строкам;
- валит проверку, если покрытие падает ниже базового порога `56%`.

Текущий базовый порог выбран по фактическому состоянию репозитория. Его задача сейчас: не дать покрытию просесть ниже текущей базы. Дальше порог нужно поднимать только вверх по мере добавления тестов.

Полезные варианты:

```bash
uv run python scripts/selfcheck.py --coverage
uv run python scripts/selfcheck.py --coverage --min-coverage 80
uv run python scripts/selfcheck.py --build
uv run python scripts/selfcheck.py --slow
uv run python scripts/selfcheck.py --hardware
uv run python scripts/selfcheck.py --slow --hardware --build
```

Рекомендуемый цикл разработки теперь такой:

1. Перед крупной правкой: `uv run python scripts/selfcheck.py --coverage`
2. После изменения runtime-логики: `uv run python scripts/selfcheck.py --coverage --build`
3. Перед проверкой на реальном железе: `uv run python scripts/selfcheck.py --coverage --hardware`

Для AI-агента это считается обязательным минимумом: сначала обновить тесты под новую функциональность, затем прогнать релевантные тесты, затем проверить coverage, чтобы не пропустить регрессию в пользовательских сценариях.

## Документация через MkDocs

В репозитории есть каркас статического сайта на MkDocs, который автоматически собирает API-справку из docstring в текущем Python-коде. Это полезно, если нужно быстро поднять GitHub Pages без ручного дублирования структуры runtime-модуля.

Подготовка зависимостей:

```bash
uv sync --group docs
```

Локальный просмотр сайта:

```bash
uv run --group docs mkdocs serve
```

Проверочная сборка:

```bash
uv run --group docs mkdocs build --strict
```

Во время сборки MkDocs автоматически запускает pre-build hook `scripts/generate_docs.py`, который читает `main.py`, `setup.py` и runtime-модули в `src/`, вытаскивает module/class/function docstring и обновляет API-страницы в каталоге `docs/`.

В репозитории также есть workflow [.github/workflows/deploy-docs.yml](.github/workflows/deploy-docs.yml), который на каждом push в `main` или `master` пересобирает сайт и публикует каталог `site/` в GitHub Pages. Это значит, что после merge ветки `docs` в `main` сайт обновится автоматически.

Для первого деплоя в самом репозитории GitHub один раз проверьте `Settings → Pages`: источник должен быть `GitHub Actions`.

## Хоткеи

По умолчанию на macOS используется:

```bash
cmd_l+alt
```

Поддерживаются комбинации из двух и более клавиш. Примеры:

```bash
uv run python main.py -k cmd_l+shift+space
uv run python main.py -k cmd_r+shift
uv run python main.py -k ctrl+alt
```

При необходимости можно включить второй хоткей для тех же действий старта и остановки записи. Это удобно, если кнопка мыши в Logitech Options или другой утилите отправляет отдельное сочетание клавиш:

```bash
uv run python main.py -k cmd_l+alt --secondary_key_combination ctrl+shift+space
```

В menu bar оба хоткея показываются отдельно:

- основной хоткей
- дополнительный хоткей

Их можно менять независимо друг от друга прямо во время работы приложения.

Все глобальные хоткеи и `Escape` теперь проходят через один runtime-dispatcher. Это важно по двум причинам:

- логика `modifier-only` сочетаний вроде `cmd_l+alt` и обычных сочетаний вроде `ctrl+shift+space` живёт в одной точке;
- при доступном `Accessibility` dispatcher пытается подавлять сработавший хоткей и не пропускать его дальше в активное приложение.

`Escape` во время записи тоже проходит через этот же dispatcher: запись отменяется, а само событие не должно уходить дальше.

Важно: сочетания вида только из modifier-клавиш на macOS сильнее зависят от системных разрешений и системного поведения клавиатуры. Если хоткей не срабатывает, сначала проверьте статусы `Accessibility` и `Input Monitoring` прямо в меню приложения и в `System Settings`.

Если `CGEventTap` не удалось поднять, приложение не переключается на неподавляющий fallback для глобальных хоткеев. В этом случае используйте menu bar и сначала восстановите системные разрешения.

## Методы вставки текста

Приложение поддерживает три метода вставки распознанного текста. Методы настраиваются в подменю «📝 Метод ввода» и применяются в порядке приоритета:

1. **CGEvent Unicode** (включён по умолчанию) — отправляет текст как Unicode-события клавиатуры через `CGEventKeyboardSetUnicodeString`. Не затрагивает буфер обмена. Работает в большинстве приложений.

2. **Accessibility API** (выключен по умолчанию) — записывает текст напрямую в сфокусированный UI-элемент через `AXUIElementSetAttributeValue`. Не затрагивает буфер обмена. Может не работать в некоторых приложениях.

3. **Буфер обмена (Cmd+V)** (выключен по умолчанию) — копирует текст в буфер обмена, отправляет Cmd+V и **восстанавливает** предыдущее содержимое буфера. Классический способ, наиболее совместимый.

Если метод выбрасывает ошибку, приложение автоматически переходит к следующему включённому методу. Если ни один метод не сработал, текст доступен в подменю «📋 История текста».

Настройки методов сохраняются между перезапусками через `NSUserDefaults`.

Туда же теперь сохраняются выбранные модель, язык, микрофон, лимит записи, хоткеи, LLM-промпт, флаг уведомления о старте записи и режим работы.

## История текста

Все распознанные тексты автоматически сохраняются в подменю «📋 История текста» (до 20 записей). Клик по элементу истории копирует полный текст в буфер обмена.

История сохраняется между перезапусками приложения через `NSUserDefaults`.

Если включить «🕶 Приватный режим» в меню приложения, новые записи истории больше не сохраняются между перезапусками: история ведётся только в памяти текущего запуска.

## Режимы работы

В меню появился переключатель `⚡ Режим работы`:

- `Обычный` — экономит память и CPU: запись идёт с более крупным буфером, а LLM после ответа выгружается из памяти.
- `Быстрый` — уменьшает задержку: запись идёт с меньшим буфером, а LLM остаётся прогретой в памяти между запросами.

Выбранный режим сохраняется между перезапусками.

## Доступы macOS

Для собранного `.app` проверьте доступы:

- `Microphone`
- `Accessibility`

Если хоткей не реагирует или текст не вставляется, дополнительно проверьте:

- `Input Monitoring`

Важно: после переноса `.app` в другую папку macOS может считать его новым приложением. В этом случае доступы иногда нужно выдать заново.

То же самое часто происходит при обновлении через простое перетаскивание новой версии `.app` из `dist` в `Applications`: macOS может сбросить доверие к `Accessibility` и `Input Monitoring`, даже если bundle identifier не менялся.

Практический совет: после первой сборки лучше перенести приложение в `/Applications`, удалить старую запись из `Accessibility` и добавить уже новую копию оттуда. Это снижает число странных кейсов с правами и старыми путями.

## Где смотреть логи

У приложения есть простые пользовательские логи:

```bash
~/Library/Logs/whisper-dictation/stdout.log
~/Library/Logs/whisper-dictation/stderr.log
```

Они полезны, если приложение запускается, но не вставляет текст, не реагирует на хоткей или не видно ошибок в интерфейсе.

Что обычно видно в логах:

- ошибки доступа к `Accessibility` или `Input Monitoring`
- проблемы с записью с микрофона
- ошибки распознавания модели
- падение вставки через CGEvent Unicode, AX API или буфер обмена
- запуск приложения с текущей моделью и хоткеем

Для просмотра в реальном времени:

```bash
tail -f ~/Library/Logs/whisper-dictation/stdout.log
tail -f ~/Library/Logs/whisper-dictation/stderr.log
```

## Техническая основа

Проект держится близко к MLX Whisper API:

- используется прямой вызов `mlx_whisper.transcribe(audio, path_or_hf_repo=...)`
- модель задается как Hugging Face repo или локальный путь
- аудио пишется через `PyAudio`
- горячие клавиши отслеживаются через `pynput`
- автоматическая вставка выполняется через настраиваемую цепочку методов: CGEvent Unicode, Accessibility API или буфер обмена с восстановлением; все распознанные тексты сохраняются в историю
- приложение в строке меню реализовано через `rumps`
