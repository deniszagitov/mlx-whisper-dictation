## Plan: Локальный голосовой агент с Obsidian

Идея реализуема на текущем проекте и на MacBook Pro M3 Max с 36 ГБ unified memory, если целиться не в «универсального автономного агента», а в локальный голосовой пайплайн Whisper → агентный раннер → локальные инструменты по заметкам → краткий ответ. Текущий код уже покрывает хоткей, запись, локальную транскрибацию и локальный MLX-LLM, но архитектурно LLM сейчас используется как короткий one-shot помощник для буфера обмена и вставки текста. Рекомендуемый путь: сначала сделать read-only агент для Obsidian с детерминированным поиском по vault и summarization, а уже потом расширять до более общего tool-calling.

**Steps**
1. Уточнить продуктовый сценарий: read-only recall по заметкам Obsidian или полноценный агент с действиями. Базовый рекомендуемый scope: пользователь говорит запрос, система локально ищет релевантные заметки, LLM кратко отвечает и при желании кладёт ответ в буфер обмена.
2. Выделить новый режим поверх существующего LLM-пайплайна, не ломая обычную диктовку и текущий LLM-hotkey. Это зависит от текущих точек входа main.py и логики transcribe_for_llm в src/transcriber.py.
3. Добавить локальный слой retrieval до LLM: сначала быстрый full-text/metadata search по markdown-файлам vault Obsidian, затем опционально гибридный поиск с embeddings. Этот шаг должен быть детерминированным и не полагаться на саму LLM для поиска по заметкам.
4. Добавить агентный раннер поверх src/llm.py: цикл «распознанный запрос → выбор инструмента/шага → локальный вызов tool → финальный ответ». На первом этапе лучше ограничить набор инструментов 2-3 read-only инструментами: search_notes, read_note, list_recent_notes.
5. Ослабить текущие ограничения короткого UI-ответа только для нового агентного режима. Сейчас проект ориентирован на 150 токенов и 180 символов ответа, что хорошо для переписывания текста, но тесно для recall по заметкам.
6. Вынести настройки нового режима в конфиг и UI: путь к vault, режим read-only, лимиты результата, выбор модели для агентного режима, отдельный prompt preset.
7. Добавить тесты регрессий для retrieval, маршрутизации инструментов, fallback-логики и для сценария «нет релевантных заметок». Это зависит от шагов 3-6.
8. Обновить README.md и пользовательские описания режима, включая локальность обработки, требования к железу и поведение буфера обмена.

**Decisions**
- Предположение: под «Obsidia» имеется в виду локальный vault Obsidian на диске, а не облачный сервис.
- Рекомендуемый первый релиз: только read-only доступ к заметкам и только локальная обработка.
- Не рекомендуется сразу строить open-ended multi-tool agent. Для пользовательского запроса вида «что у меня было про рынок» лучше сначала сделать retrieval-first пайплайн, где поиск по заметкам выполняется кодом, а LLM только суммирует найденное.
- Текущая дефолтная модель Qwen 3.5 4B в 6-bit MLX должна хватить для прототипа recall/summarization и простого роутинга, но для более строгого tool-calling и сложного многошагового поведения вероятно потребуется более надёжная локальная модель или жёстче детерминированная оркестрация.

**Integration options researched**
- MCP — лучший интерфейс именно для агентного раннера, когда LLM должна видеть типизированные tools и вызывать их по протоколу; это хороший слой абстракции, но сам по себе MCP не решает retrieval и добавляет ещё один интеграционный слой.
- Obsidian Local REST API — наиболее богатый интерфейс для работы с vault из локального приложения: даёт HTTPS API с API key, CRUD по файлам, targeted patch, full-text search, Dataview/JsonLogic search, list/execute commands, open note in UI.
- Obsidian URI — лёгкий способ открыть поиск/заметку/создать заметку, но слаб как backend retrieval interface; подходит скорее для UI handoff, чем для поиска контекста для LLM.
- Прямой поиск по markdown-файлам vault — самый дешёвый и предсказуемый локальный вариант по CPU/RAM и по отказоустойчивости; особенно хорош для read-only retrieval-first архитектуры.
- Git как интерфейс к Obsidian vault — полезен как дополнительный источник истории изменений и provenance, но плох как основной слой online-retrieval для быстрых пользовательских запросов.

**Further Considerations**
1. Источник заметок: только один vault Obsidian или несколько. Рекомендация: начать с одного vault и явного пути в настройках.
2. Retrieval: lexical search или embeddings. Рекомендация: сначала lexical/hybrid без тяжёлой векторной инфраструктуры; embeddings оставить второй фазой.
3. Ответ пользователю: только буфер обмена/уведомление или отдельное окно результата. Рекомендация: для recall по заметкам предусмотреть более длинный вывод, иначе ограничение короткого LLM-ответа будет мешать.
